5. **No auto-promotion** — eval findings and pending entries never become L-series. The knowledge graph write path is the real fix, but until then, nothing bridges the gap.

PROMPT 4 OF 4: Backtest, Tune, and Verify End-to-End
Estimated: ~10K active tokens
===========================================================================

## Problem

Prompts 1-3 built the store, wired writes, and wired reads. But we don't know if the FTS5 queries return RELEVANT results. A query for "Market Map widget with DuckDB submarket aggregation" should return learnings about DuckDB patterns, map rendering, and submarket data — not learnings about authentication or git branching.

Without backtesting, we're trusting that keyword matching works. The eval sidecar data (113 JSONs) is ground truth for what actually went wrong per feature. We can measure whether the store would have surfaced the right knowledge.

## What to build

A `backtest()` function in `knowledge_store.py` that:

1. Takes an evals directory and a specs directory (or project dir).
2. For each eval JSON:
   - Extract the feature name from the eval.
   - Find the matching feature spec in `.specs/features/` if it exists, otherwise use the feature name.
   - Run `query_relevant(conn, feature_text, limit=20)`.
   - Extract the `agent_eval` fields from the eval JSON (scope_assessment, framework_compliance, repeated_mistakes, eval_notes).
   - Score relevance: do any returned learnings' tags or body text overlap with the issues the eval found? A learning tagged "DuckDB" is relevant if the eval_notes mention DuckDB. A learning about "scope_assessment: sprawling" is relevant if the eval found sprawling scope.
3. Aggregate: precision (relevant returns / total returns) and recall (issues with matching learning / total issues) across all features.
4. Return a structured report: per-feature hits/misses, aggregate precision/recall, and the worst misses (features where the eval found serious issues but the store returned nothing relevant).

Also: a `tune_weights()` function or guidance on FTS5 column weights. FTS5 supports `rank` with column weights — e.g., tags should be weighted higher than body text because tags are curated keywords. The backtest results tell you what to adjust: if precision is low, increase tag weight (fewer but more targeted results); if recall is low, increase body weight (broader matching).

## Where to find the answers

- `py/auto_sdd/lib/knowledge_store.py` (from Prompts 1-3): `query_relevant()`, `init_store()`.
- `logs/compstak-sitdeck/evals/eval-*.json` — ground truth. Fields: `agent_eval.scope_assessment`, `agent_eval.framework_compliance`, `agent_eval.repeated_mistakes`, `agent_eval.eval_notes`.
- `~/compstak-sitdeck/.specs/features/` — feature specs (13 exist; for features without specs, use the feature name from the eval JSON).
- FTS5 documentation: column weights in `rank` function, e.g. `SELECT *, rank FROM learnings_fts WHERE learnings_fts MATCH ? ORDER BY rank` — the `rank` function accepts a weights array matching column order.

## Scope

- Modify: `.handoff.md` — update the broken connections table to show fixed state for all 5 rows

PROMPT 5 OF 5: Continuous Retrieval Optimization Loop
Estimated: ~10K active tokens
===========================================================================

## Problem

After Prompt 4's one-time backtest, the system has calibrated FTS5 weights against 113 eval findings. But it has no mechanism to detect retrieval quality degradation as new learnings accumulate, new projects run, or the eval distribution shifts. Without continuous measurement, the store will silently rot — returning increasingly irrelevant results as the corpus grows and no one notices.

## What to build

Three pieces that close the optimization loop:

**1. Retrieval logger.** In `prompt_builder.py`, wherever `query_relevant()` is called during `build_feature_prompt()`: log the query text, the returned learning IDs + BM25 ranks, the feature name, and the timestamp to `<project_dir>/.sdd-state/retrieval_log.jsonl`. One line per feature build. This is the "what did we inject" record.

**2. Post-eval comparison.** In `eval_sidecar.py`, after each eval JSON is written: read the most recent retrieval log entry for that feature. Compare the tags/keywords from returned learnings against the issues the eval found (scope_assessment, framework_compliance, repeated_mistakes, eval_notes). Score each returned learning as hit (relevant to an actual issue) or miss (irrelevant). Append hit/miss counts to the same retrieval log entry. This is the "was what we injected actually useful" record.

**3. Periodic re-tune trigger.** In `build_loop.py`, at the end of each campaign (after post-campaign verify, before auto-QA): read `retrieval_log.jsonl`, compute rolling precision and recall over the last N features (N = number of features in the just-completed campaign). If precision < the threshold established in Prompt 4's initial backtest OR recall < that threshold: re-run the `backtest()` function from Prompt 4 with the current store, auto-adjust FTS5 column weights, log the adjustment. This is mechanical — no agent, no human.

## Where to find the answers

- `py/auto_sdd/lib/prompt_builder.py`: `build_feature_prompt()` — find where `query_relevant()` is called (from Prompt 3). Add the retrieval log write immediately after.
- `py/auto_sdd/scripts/eval_sidecar.py`: find where eval JSON is written to disk. Add retrieval log comparison after.
- `py/auto_sdd/scripts/build_loop.py`: find where `_post_campaign_verify()` is called (two places, ~line 797 and ~837). Add the periodic re-tune check after verify, before auto-QA.
- `py/auto_sdd/lib/knowledge_store.py` (from Prompt 1): `query_relevant()` — understand the return format (list of dicts with id, title, tags, rank). Also `backtest()` from Prompt 4 — understand its precision/recall output. Add a `retune_weights(conn, target_precision, target_recall, retrieval_log_path)` function that reads the log, computes metrics, adjusts FTS5 column weights if below thresholds, and returns what was changed.
- FTS5 column weights: `SELECT *, rank FROM learnings_fts('query', rank_weights)` — the rank function accepts a weights string like `'1.0 2.0 3.0'` matching the column order in the FTS5 table. Adjusting these changes which columns contribute more to ranking.
- `.sdd-state/retrieval_log.jsonl` — does not exist yet. Schema per line: `{"timestamp", "feature_name", "query_text", "returned_ids": [...], "ranks": [...], "eval_hits": N, "eval_misses": N, "precision": float, "recall": float}`. Early entries (before eval runs) will have null for eval_hits/misses/precision/recall — the eval comparison fills them in later.
