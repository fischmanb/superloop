### 3a. `learnings/constraints/go.md`

Title: "Go Transitive Boundary Violations"

Patterns (12):
1. `internal/` package visibility — code outside parent tree can't import internal/ packages
2. Circular package imports — Go compiler rejects A→B→A outright
3. Build tag mismatch — `//go:build linux` in file included on darwin/windows build
4. CGO linking — importing C-dependent package without C build toolchain installed
5. `replace` directive scope — go.mod `replace` doesn't propagate to modules that depend on yours
6. Vendor vs module mode — `vendor/` dir exists but `go mod vendor` is stale or `-mod=vendor` not set
7. `init()` side effects — package-level init() runs on import; transitive import triggers unexpected initialization
8. Unexported type in public API — returning `*internalType` from exported function; compiles locally but consumers can't use the type
9. Test file leakage — `_test.go` importing test-only packages that bleed into shared fixtures
10. Plugin ABI — `plugin.Open` requires exact Go version + build flag match between plugin and host
11. `//go:embed` path restriction — paths relative to source file only, no `..`, no symlinks, no cross-module
12. Package-level variable races — shared mutable state across goroutines, caught by `-race` flag but not default build

### 3b. `learnings/constraints/rust.md`

Title: "Rust Transitive Boundary Violations"

Patterns (12):
1. Missing feature gate — using `tokio::fs` but Cargo.toml only enables `features = ["rt"]`, not `"fs"`
2. Async runtime mixing — tokio primitives in async-std project (or vice versa), may compile but panics at runtime
3. `no_std` boundary — `std` type used in `#![no_std]` crate's dependency chain
4. Edition mismatch — workspace members on different editions causing macro behavior differences
5. Proc macro crate boundary — proc macros must live in separate `crate-type = ["proc-macro"]` crate, can't import runtime types from parent
6. Unsafe dependency chain — safe wrapper calling unsafe transitively, `#[forbid(unsafe_code)]` catches it late
7. Orphan rule — implementing foreign trait on foreign type (E0117), must use newtype wrapper
8. Target-specific `cfg` — `#[cfg(target_os = "linux")]` code imported unconditionally in cross-platform build
9. Dev-dependency in non-test — `[dev-dependencies]` crate used in `src/`, compiles in `cargo test` but fails on `cargo build`
10. Workspace version conflicts — two workspace members pulling different incompatible versions of same transitive dep
11. `build.rs` feature dependency — build script depending on features only enabled in certain profiles
12. `Send`/`Sync` boundary — type used across thread boundary missing trait bound, error surfaces deep in generic call chain

### 3c. `learnings/constraints/python.md`

Title: "Python Transitive Boundary Violations"

Patterns (12):
1. Circular import at module level — `from a import X` in b and `from b import Y` in a → ImportError
2. Undeclared dependency — `import pandas` works locally, fails on clean install (not in pyproject.toml)
3. Import-time side effects — module does DB connect or network call at import; transitive import triggers it unexpectedly
4. Async/sync mixing — calling `await` from sync context or sync blocking call inside async loop
5. Relative import outside package — `from ..utils import X` when file isn't in a proper package (missing `__init__.py` or flat layout)
6. `__init__.py` re-export chains — A's init imports from B, B imports from C, C imports from A → hidden circular
7. Optional dependency imported unconditionally — `import scipy` in main code but scipy is in `[optional-dependencies]`
8. Platform-specific module — `import fcntl` on Windows or `import winreg` on Linux → ImportError
9. Type stub mismatch — `py.typed` marker present, stubs exist, but stubs don't match actual runtime signatures
10. Namespace package collision — two installed packages claiming same namespace prefix, import resolves to wrong one
11. `sys.path` manipulation — runtime path hacking in production code breaks install-time resolution for downstream consumers
12. Test fixture → production → test loop — test helper imports prod code that imports test-only dep transitively

### 3d. `learnings/constraints/nextjs.md`

Title: "Next.js / React SSR Transitive Boundary Violations"

Patterns (10):
1. `"use client"` → server module (L-00012) — client component transitively imports postgres/fs/crypto via shared utility
2. Server component → client-only library — importing `framer-motion`, `window` APIs, or browser-only packages in server component
3. `"use server"` re-export leak — server action exported from barrel file that client imports
4. Dynamic import with `ssr: false` — still pulls server deps into webpack build graph even if not executed
5. Middleware edge runtime — middleware.ts imports package using Node built-ins (fs, crypto, stream); edge runtime doesn't support them
6. `next/headers` / `next/cookies` outside server context — used in shared utility imported by client component
7. Shared types file pulling implementation — `types.ts` imports from `db/index.ts` for type inference, pulls server dep transitively
8. Missing `"use client"` — component uses `useState`/`useEffect` but no directive; error says "hooks can only be called in client component"
9. Server action non-async — functions in `"use server"` file that aren't async; Next.js requires server actions to be async
10. `generateMetadata` / `generateStaticParams` — these server-only functions importing utility that transitively reaches client bundle

Title: "TypeScript / JavaScript General Transitive Boundary Violations"

Patterns (10):
1. Node built-in in browser bundle — `fs`, `path`, `crypto`, `child_process` imported in code targeting browser bundler
2. ESM/CJS boundary — `require()` in ESM module or named import from CJS-only package; behavior varies by bundler
3. `devDependency` in production — test/build tool imported in `src/`, works in dev but fails in production build
4. Barrel file side effects — `index.ts` re-exporting everything defeats tree-shaking, pulls all modules into bundle

9. Dynamic `require()` — bundler can't statically analyze `require(variable)`, silently excludes module or throws at runtime
10. Ambient module masking — `declare module 'x'` in `.d.ts` file makes TypeScript happy but package isn't actually installed; compiles, crashes at runtime

## Step 5: Build Detection + Injection in prompt_builder.py

Add these functions to `prompt_builder.py`. Place them ABOVE `build_feature_prompt`.

### 5a. `detect_project_traits(project_dir: Path) -> frozenset[str]`

Scans the project directory and returns a frozenset of trait strings. Detection rules:

| Check | Trait(s) added |
|-------|---------------|
| `next.config.*` exists (glob) | `nextjs` |
| Any `.tsx`/`.jsx` file's first 5 lines contain `"use client"` | `nextjs` (if not already), plus this confirms SSR boundary |
| `go.mod` exists | `go` |
| `internal/` directory exists (direct child of project_dir) | `go` |
| `Cargo.toml` exists | `rust` |
| `pyproject.toml` or `setup.py` exists | `python` |
| `tsconfig.json` exists | `typescript-general` |
| `package.json` exists | `typescript-general` |

Implementation constraints:
- Do NOT read entire directories recursively for most checks — just check file/dir existence.
- For `"use client"` scanning: find first 200 `.tsx`/`.jsx` files via `project_dir.rglob()`, read only first 5 lines each, stop on first match. Exclude `node_modules`, `.git`, `dist`, `build`, `.next`, `__pycache__`, `target` from rglob.
- Return `frozenset` for immutability.
- Each trait string MUST match a constraint filename (minus `.md`).

### 5b. `select_constraint_files(traits: frozenset[str], constraints_dir: Path) -> list[Path]`

Returns list of constraint file paths where the filename stem is in `traits`, plus always include `cross-language.md` if at least one trait matched.

Example: traits = `{"nextjs", "typescript-general"}` → returns paths for `nextjs.md`, `typescript-general.md`, `cross-language.md`.

### 5c. `format_constraints_block(constraint_files: list[Path]) -> str`

Reads each file, returns concatenated content with `---` separators between files. No added headers — the caller adds the section header.
